Root Cause 분석 Fail-soft UX 설계 CTO 협업

배송 오류 문의 반복 → Root Cause 규명 → CTO 재제안 → 문의 0건

고객 문의가 반복될 때, 빠른 응대 대신 원인을 파고들었습니다. 반년 전 거절됐던 UX 개선 아이디어를 데이터와 함께 CTO에게 재제안해 당일 구현·배포까지 이어낸 사례입니다.

회사(주)초록마을
기간2024.12 – 재직 중
역할CS운영 / 서비스 기획 병행
문제 정의

고객이 아니라 서비스 구조에 문제가 있었습니다

배송 가능일이 다른 상품이 장바구니에 함께 담기면 주문이 차단되는 상황에서, 고객에게는 이유가 전혀 안내되지 않아 오류로 인식되는 문제가 있었습니다. 팀은 "상품을 하나씩 빼보세요"라는 안내로 대응했지만, 일 6~10건의 동일 문의가 반복됐습니다.

반년 전, 이 문제를 처음 발견해 개발팀에 개선을 제안했습니다. 하지만 기술 제약을 이유로 미뤄진 채 동일한 문의가 계속 들어오고 있었습니다. 반복 문의는 고객의 문제가 아니라 서비스 구조의 문제였습니다.

실행 — Root Cause 재정의 후 MVP로 재제안

기술 제약 안에서 MVP 재설계 → CTO에게 데이터로 재제안 → 당일 구현

"기술 제약으로 모든 예외 케이스를 처리하기 어렵다"는 개발팀 답변에서 멈추지 않았습니다. 문제를 다시 분해했고, API 응답값으로 판단 가능한 조건만 남긴 최소 실행 단위(MVP)로 재설계했습니다. 반년 전 거절됐던 이슈를 반복 문의 데이터와 함께 다시 꺼내 CTO에게 직접 재제안했습니다.

CTO가 제안한 "주의상품" 서버 태깅 방식을 기반으로, 장바구니 내 상품별 안내 문구와 재사용 가능한 라벨 구조를 직접 기획했습니다. "⚠️ 화요일에만 도착 가능한 상품이에요"처럼 고객이 주문 불가 이유를 즉시 이해할 수 있도록 Fail-soft UX를 설계했고, 제안부터 개발·배포까지 당일 완료됐습니다.

배포 다음날에는 웹에서 줄바꿈 깨지는 이슈를 직접 발견해 수정을 제안하고, 동일한 구조의 유사 상품에도 주의상품 태깅 확대를 요청했습니다. 이후 1+1·2+1 프로모션 상품으로도 확대 적용됐습니다.

증빙

실제 구현 화면 및 제안 과정

1. 주의상품 시스템 적용 후 장바구니 화면
"⚠️ 화요일에만 도착 가능한 상품이에요. 나누어 주문이 필요할 수 있어요." 기존에 이유 없이 차단되던 것이 실제 사유로 전환됐습니다. 이후 상품운영파트가 동일한 subtext 구조를 1+1·2+1 프로모션 안내에도 자체 확산 적용했습니다 — 요청 없이 전파됐다는 것은 이 구조가 진짜 문제를 해결했다는 신호입니다.
2. CTO에게 직접 제안한 슬랙 DM
반년 전 거절됐던 이슈를 데이터와 함께 재제안했습니다. "구분이 된다면 문구만 다르게 보여주는 MVP로 가능할까요?"라는 질문에 CTO가 즉시 "주의상품 방식으로 해볼게요"하고 개발팀을 태그했습니다. 제안부터 배포까지 당일 완료됐습니다.
3. 배포 다음날 — 품질 개선 제안 + 유사 상품 확대 요청
배포 직후 웹에서 줄바꿈 깨지는 이슈를 발견해 한 문장 포맷으로 수정을 제안했고, 다음날 동일한 구조의 상품에도 주의상품 태깅 확대를 직접 요청했습니다.
